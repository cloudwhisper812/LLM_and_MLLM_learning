




# QLoRA: Efficient Finetuning of Quantized LLMs
> 它是用来换空间的，不是用来提速的。反量化的过程带来了额外的计算开销。QLoRA 的训练速度往往比纯 BF16 的 LoRA 还要稍微慢一点，但它让你能在一张卡上跑起以前需要八张卡才能跑的模型。


## 1. 核心要点与实验结论 (TL;DR)

### 1.1 NF4 (4-bit NormalFloat)
* **直觉与做法**：模型参数大多服从零均值的正态分布，如果用传统的 INT4 均匀切分，落在密集区的权重会有很大的量化误差。所以 NF4 改用正态分布的 quantile（分位数）来做切分，让接近 0 的地方切分更密集。
* **工程 Hack**：离线打表，不动态算分位数，而是针对标准正态分布预计算 16 个点。非对称零点：左边切 8 份，右边切 9 份，强制凑出一个绝对的 0.0 用于处理 Padding 等特殊值。
* **计算过程**：量化时，首先将参数 normalize 到 -1 到 1 之间，然后跟预先算好的 16 个 bin 的中间数字去找最近邻（Nearest Neighbor）。

### 1.2 Double Quantization (双量化)
* **直觉与做法**：第一遍量化完，每个 block 会产生一个 scaling factor，这些 factor 默认是 FP32 的，积少成多还是很吃显存。DQ 就是把这些 factor 再做一次 8-bit 量化。
* **收益**：平均下来，每个参数额外占用的管理显存从 0.5 bit(通常64个参数一组量化， 又一个fp32的factor， 32/64=0.5) 降到了 **0.127 bit**。在 65B 模型上能硬生生抠出约 3GB 显存，留给 Batch Size。

### 1.3 Paged Optimizer (分页优化器)
* **直觉与做法**：利用 NVIDIA 的 Unified Memory。显存快爆的时候（比如遇到特别长的 Sequence），把暂时不用的优化器状态（Optimizer States）踢到 CPU 内存里，等反向传播算到它了再拷回来。
* **收益**：没有魔法，纯粹是个防 OOM 的系统级兜底机制，平时基本不触发，触发了速度也基本不掉。

### 1.4 实验总结
* **结论**：只要在**所有的线性层（All-linear）**都打上 LoRA，4-bit NF4 基座微调出来的效果，完全等价于 16-bit 全参微调。
* **战果**：跑出了 Guanaco 65B，单卡 24 小时微调完，实战盲测表现打平了 ChatGPT (GPT-3.5)。

## 2. 对现代训练框架与工程落地的影响

### 2.1 训练显存的宏观账本：到底省在了哪？
* **全参微调（FFT）的显存噩梦（以 7B 为例）**：
  * **模型权重（FP16）**：约 14GB。
  * **梯度（FP16）**：约 14GB。
  * **Adam 优化器状态**：这是真正的吃存大户，需要存 FP32 的 Master Weights（28GB）、动量 Momentum（28GB）和方差 Variance（28GB），共计 84GB。
  * **总计**：啥数据都还没跑，模型+优化器就已经吃掉了 **112GB** 显存。
* **QLoRA 的降维打击**：
  * **权重**：基座权重压成 NF4，7B 模型只需约 4GB。
  * **梯度与优化器**：由于基座冻结，梯度和 Adam 状态**只针对 LoRA 那一丁点参数**（通常仅需几百 MB）。
* **补充：激活值（Activations）的解法**：前向传播缓存的激活值也是显存杀手（尤其在长 Context 下）。业界的标准做法是配合 **Gradient Checkpointing（梯度检查点）** 技术，用时间换空间（反向传播用到时再重算前向），与 QLoRA 结合使用，彻底把显存需求打下来。

### 2.2 核心直觉：知识保留与“短平快”的任务适配
* **底座知识极其鲁棒**：Pretrain 阶段用海量数据和算力砸出来的“世界知识”是非常稳固的。实验证明，即使把精度用 NF4 压榨到 4-bit，模型原本沉淀的知识（常识、逻辑、语言能力）也并没有丢失。
* **微调的本质**：既然知识库还在，我们就不需要兴师动众地去全量修改它。利用 QLoRA 挂载轻量级的 Adapter，能够以“短平快”的方式，低成本、高效率地将模型的能力激发出来，快速对齐（Alignment）到特定的下游任务（如指令跟随、垂直领域问答）上，性价比极高。

### 2.3 工业界踩坑：训完直接推？速度会翻车
* **现实情况**：如果你拿 QLoRA 训完的模型直接上线推理，速度其实比 16-bit 全量模型**还要慢**。因为 GPU 没有原生 NF4 算子，推理时每次过线性层都要做**实时反量化（On-the-fly Dequantization）**，严重拖累计算效率。
* **主流落地解法**：训完之后，离线把 NF4 基座反量化回 FP16，把 LoRA 的权重矩阵相加（融卡）。得到一个标准的 FP16 模型后，再用 **GPTQ / AWQ** 重新量化成 INT4 去做部署（因为 INT4 有极其优化的推理算子）。
